# 失物招领系统 - 项目答辩文档

## 一、项目概述

### 1.1 项目名称
**校园失物招领平台 (LostAndFound)**

### 1.2 项目背景与意义
在校园生活中，物品丢失和拾获是高频事件。传统的失物招领方式（如张贴纸质公告、校园论坛发帖）存在信息传播慢、匹配效率低、缺乏实时交互等痛点。本系统旨在打造一个**一站式、智能化、实时互动**的校园失物招领管理平台。

### 1.3 核心功能
| 功能模块 | 描述 |
|---------|------|
| **失物/拾物发布** | 用户可发布丢失或拾到的物品信息，支持图片上传、位置标注、分类标签 |
| **智能搜索与匹配** | 基于物品特征的关键词搜索 + 自动推荐匹配算法 |
| **实时聊天** | 失主与拾取者之间实时沟通，协商归还事宜 |
| **好友系统** | 添加好友、备注管理、私聊、删除好友 |
| **认领流程** | 完整的申请→审核→确认→归还闭环流程 |
| **消息通知** | 站内信通知（好友请求、审核结果等） |
| **个人中心** | 个人资料管理、头像裁剪上传、发布记录、密码修改 |
| **管理后台** | 用户管理、物品审核、公告发布、数据统计、地图点位管理 |

## 三、核心设计详解

### 3.1 JWT 认证机制

#### 为什么选择 JWT？

传统 Session 认证在前后端分离架构下存在问题：
- **Session 存储在服务器端**：增加服务器内存压力
- **跨域问题**：前端(8081) 和后端(8080) 不同源，Cookie 无法直接共享
- **集群部署困难**：多台服务器需要 Session 同步（Redis/Sticky Session）

**JWT (JSON Web Token)** 的优势：
```
┌────────────────────────────────────────────────────┐
│                    JWT Token 结构                     │
│                                                      │
│  Header.Payload.Signature                            │
│                                                      │
│  ┌──────────┐ ┌──────────────────────┐ ┌─────────┐ │
│  │ {"alg":  │ │ {"sub":"zhangsan",  │ │ HMAC-   │ │
│  │  "HS256"}│ │  "iat":1700000000,  │ │ SHA256  │ │
│  │          │ │  "exp":1700086400}  │ │ 签名    │ │
│  └──────────┘ └──────────────────────┘ └─────────┘ │
│     ↑ Base64           ↑ Base64           ↑ 密钥    │
└────────────────────────────────────────────────────┘
```

#### 本项目 JWT 认证流程

```
┌────────┐     ① POST /api/auth/login      ┌──────────┐
│  前端   │ ──────────────────────────────> │  后端    │
│        │                                    │          │
│        │     ② 返回 accessToken           │ 验证账号  │
│        │ <─────────────────────────────── │ 密码     │
│        │        refreshToken              │ 生成JWT  │
│        │        user对象                 │          │
│        │                                    │          │
│ ③ 存储  │                                    └──────────┘
│ token到 │
│sessionStorage
│        │
│ ④ 后续请求 Header: Bearer {token}      ┌──────────┐
│        │ ──────────────────────────────> │  后端    │
│        │                                    │ JwtRequest│
│        │     ⑤ 返回数据 / 401未授权      │ Filter   │
│        │ <─────────────────────────────── │ 解析Token│
│        │                                    │ 校验过期 │
│        │  401时尝试用refreshToken刷新      │ 设置SecurityContext│
│        │                                    │          │
│        │                                    └──────────┘
└────────┘
```

**关键设计点：**
- **双 Token 机制**：`accessToken`（短期，15分钟）+ `refreshToken`（长期）
- **自动刷新**：Axios 拦截器捕获 401 错误，自动调用 `/auth/refresh` 获取新 Token 并重试请求
- **无状态**：服务器不存储 Session，天然支持水平扩展

### 3.2 RBAC 权限控制模型

#### 什么是 RBAC？
**RBAC (Role-Based Access Control)** = 基于角色的访问控制

核心思想：**用户 → 角色 → 权限**

```
用户(User) ──N:N──> 角色(Role) ──1:N──> 权限(Permission)
```

#### 后端 SecurityConfig 实现

```java
// 核心配置伪代码
http.authorizeRequests()
    // 公开接口（无需登录即可访问）
    .antMatchers("/api/auth/**", "/static/**", 
                 "/api/lost/**", "/api/found/**",

                 "/ws/**").permitAll()

    .anyRequest().authenticated();
```

#### 前端路由守卫

```typescript
router.beforeEach((to, from, next) => {
  const token = store.state.auth.token
  const meta = to.meta
  
  // 1. 需要登录但没token -> 跳转登录页
  if (meta.auth && !token) return next('/login')
  
  // 2. 需要特定角色 -> 检查用户角色
  const requiredRoles = meta.roles
  if (requiredRoles && user) {
    const hasRole = user.roles.some(r => requiredRoles.includes(r.code))
    if (!hasRole) return next('/login')
  }
  
  next()
})
```

#### 数据库中的角色表设计

```sql
-- 用户表
CREATE TABLE users (
  id BIGINT PRIMARY KEY AUTO_INCREMENT,
  username VARCHAR(64),
  password_hash VARCHAR(255),  -- BCrypt 加密存储
  -- ... 其他字段
);

-- 角色表
CREATE TABLE roles (
  id BIGINT PRIMARY KEY AUTO_INCREMENT,
  code VARCHAR(64),     -- ROLE_STUDENT / ROLE_MANAGER / ROLE_ADMIN
  name VARCHAR(64)      -- 学生 / 负责人 / 管理员
);

-- 用户-角色关联表（多对多）
CREATE TABLE user_roles (
  user_id BIGINT,
  role_id BIGINT

```

**为什么用多对多？**
- 一个用户可以同时拥有多个角色（如管理员也是学生）
- 方便未来扩展新角色（如"志愿者"、"教师"）

### 3.3 为什么使用 WebSocket 而非 HTTP 实现实时聊天？

这是答辩中**最高频被问到的问题**之一，以下是详细对比分析：

#### HTTP 轮询 vs WebSocket 对比

```
┌─────────────────────────────────────────────────────────────┐
│                    HTTP 轮询模式 (Polling)                   │
│                                                             │
│  客户端 ──请求──> 服务器 ──响应──> 客户端 (有消息/无消息)    │
│     │                                                    │
│     ├─ 请求 ──> 服务器 ──响应──> 无新消息                  │
│     │                                                    │
│     ├─ 请求 ──> 服务器 ──响应──> 无新消息                  │
│     │                                                    │
│     ├─ 请求 ──> 服务器 ──响应──> 有新消息!                │
│     │                                                    │
│     ├─ 请求 ──> 服务器 ──响应──> 无新消息                  │
│     │        (即使没有消息也要不断请求)                    │
│     └─ ...                                               │
│                                                             │
│  缺点:                                                    │
│  • 浪费带宽: 大量无效请求                                   │
│  • 延迟高: 最坏情况等待一个轮询周期才能收到消息             │
│  • 服务器压力大: 每秒可能收到数千个请求                      │
└─────────────────────────────────────────────────────────────┘

┌─────────────────────────────────────────────────────────────┐
│                  WebSocket 全双工模式                        │
│                                                             │
│  客户端 <────────────────── TCP连接 ─────────────────> 服务器 │
│     │                        ↑ 持久连接                    │
│     │                        │ (握手后升级为WS协议)         │
│     │                                                    │
│     │ ←──── "你好!" (服务器主动推送，无需客户端请求) ──│
│     │                                                    │
│     │ ──→ "在吗？" (客户端随时发送) ──────────────────→│
│     │                                                    │
│     │ ←──── "在的，东西在你那吗？" (即时到达) ─────────│
│     │                                                    │
│  优势:                                                    │
│  • 低延迟: 消息即时送达，毫秒级                             │
│  • 省资源: 只在有消息时才传输数据                           │
│  • 服务端推送: 服务器可以主动向客户端发送消息               │
│  • 连接复用: 一次握手，持续通信                             │
└─────────────────────────────────────────────────────────────┘
```

#### 技术原理

**WebSocket 握手过程（HTTP 升级）：**
```
客户端                              服务器
  │                                   │
  │ ─── GET /ws HTTP/1.1 ────────>  │
  │     Upgrade: websocket         │
  │     Connection: Upgrade        │
  │     Sec-WebSocket-Key: xxx     │
  │                                   │
  │ <── 101 Switching Protocols ──│
  │     Sec-WebSocket-Accept: yyy  │
  │                                   │
  ════════ 升级完成!现在是WebSocket连接 ════════
  │                                   │
  │ <── 二进制帧: "你好" ─────────> │  (TCP全双工通道)
  │ <── 二进制帧: "在的" ─────────│
  │                                   │
```

#### 本项目的 WebSocket 实现

```java
@Configuration
@EnableWebSocket
public class WebSocketConfig implements WebSocketConfigurer {
    
    @Override
    public void registerWebSocketHandlers(WebSocketHandlerRegistry registry) {
        // 注册聊天处理器，端点 /ws/chat
        registry.addHandler(chatWebSocketHandler(), "/ws/chat")
            .setAllowedOrigins("*");  // 允许跨域
    }
}
```

```java
@Component
public class ChatWebSocketHandler extends TextWebSocketHandler {
    
    // 维护在线用户的 WebSocket Session 映射
    private final Map<Long, WebSocketSession> onlineSessions = new ConcurrentHashMap<>();
    
    @Override
    public void afterConnectionEstablished(WebSocketSession session) {
        // 新连接建立，保存 session
        Long userId = extractUserId(session);
        onlineSessions.put(userId, session);
    }
    
    @Override
    protected void handleTextMessage(WebSocketSession session, TextMessage message) {
        // 收到消息 -> 解析 -> 保存数据库 -> 推送给接收方
        Long receiverId = parseReceiver(message);
        WebSocketSession receiverSession = onlineSessions.get(receiverId);
        if (receiverSession != null && receiverSession.isOpen()) {
            receiverSession.sendMessage(message);  // 主动推送！
        }
    }
}
```

#### 前端 WebSocket 连接

```typescript
// 轮询增强: 用定时器定期拉取新消息 + WebSocket 实时推送
const pollingTimer = setInterval(async () => {
  if (replyVisible.value && replyTarget.value) {
    await loadHistory(true);  // 每2秒刷新一次聊天记录
  }
}, 2000)
```

**总结：选择 WebSocket 的原因**
1. **实时性要求高**：聊天场景需要消息即时送达，HTTP 轮询延迟不可接受
2. **节省资源**：避免每秒数百次无效 HTTP 请求
3. **用户体验好**：消息"秒收"，类似微信的聊天体验
4. **技术成熟度**：HTML5 标准，浏览器原生支持，Spring Boot 内置支持

## 四、核心功能实现亮点

### 4.1 手写 Canvas 图片裁剪器（零依赖）

**为什么不使用 cropperjs/vue-cropper？**
- 减少第三方依赖包体积（~50KB gzip）
- 完全可控的定制化能力
- 学习价值高，展示前端基础功底

**核心技术点：**
- `getBoundingClientRect()` WYSIWYG 坐标映射
- CSS `box-shadow: 0 0 0 9999px` 实现圆形遮罩暗角
- Canvas `drawImage()` 九参数裁剪导出高清图片
- 同时支持 Mouse + Touch 事件，兼容移动端

### 4.2 智能匹配算法

基于物品特征的文本相似度匹配，帮助用户快速找到可能的失物/拾物对应关系。

### 4.3 多层级图片处理链路

```
原始图片 → 前端 Canvas 裁剪(400×400 JPEG) → 后端 Thumbnailator 压缩(600×600) → 存储
         ↓                    ↓
    用户头像上传          物品图片上传(原图+缩略图)
```

## 七、项目总结与创新点

### 7.1 技术创新点

| 创新点 | 实现方式 | 价值 |
|--------|---------|------|
| **零依赖图片裁剪** | 纯手写 Canvas 引擎 | 减少依赖、完全可控 |
| **双 Token 认证** | access + refresh token | 安全性与便利性兼顾 |
| **混合通信模型** | WebSocket + 轮询兜底 | 保证实时性的同时确保可靠性 |
| **智能匹配** | 文本相似度算法 | 提升物品找回率 |
| **RBAC 细粒度权限** | 三角色 + 接口级控制 | 满足不同身份需求 |

### 7.2 可扩展性设计

- **水平扩展**：JWT 无状态，后端可多实例部署
- **模块化解耦**：Controller → Service → Mapper 分层清晰
- **配置外部化**：application.yml 集中管理数据库、文件路径等
- **API 版本预留**：URL 设计考虑了版本前缀 `/api/v1/`
